# Supabase Cashflow Tracking And Tableau Deployment Plan

Related issue: `stock-analysis-8hb`

## Summary

Keep the monthly deposit assumption as a research/backtesting input, but stop using it as the live account cash source for production recommendations. Supabase should become the source of truth for actual account cashflows, account snapshots, recommendation history, and materialized performance metrics. Tableau remains the visualizer over curated outputs; it should not be the primary place where deposits are entered.

The main product flow should be:

1. Register real deposits, withdrawals, fees, and dividends in Supabase on their actual dates.
2. Import or register a current account snapshot with holdings, cash, and total value.
3. Run `run-one-shot` after market data is available.
4. Store recommendation runs and line items back to Supabase.
5. Export/publish Tableau-ready marts that show recommendations, actual deposits, account returns, and a same-cashflow SPY benchmark.

## Current State

- The configured production model is already `e8-scale-0p5-contribution-aware-v1`, and the main flow applies `forecast.ml_score_scale: 0.5`.
- `src/stock_analysis/config.py` has `ContributionConfig.monthly_deposit_amount`, `deposit_frequency_days`, `deposit_start_date`, and `rebalance_on_deposit_day`.
- `src/stock_analysis/backtest/runner.py` correctly treats the monthly deposit amount as a scheduled backtest contribution and computes cashflow-aware metrics.
- `src/stock_analysis/pipeline/one_shot.py` still passes `config.contributions.monthly_deposit_amount` to `build_rebalance_context(...)` as the immediate live contribution amount.
- `src/stock_analysis/portfolio/rebalance.py` can already model a post-deposit rebalance, but it needs an unambiguous live cash source to avoid double-counting cash already present in a holdings snapshot.
- `src/stock_analysis/tableau/dashboard_mart.py` produces a single recommendation-oriented mart. It includes contribution fields, but it does not expose a cashflow ledger, account return history, recommendation history, or same-cashflow SPY tracking.
- `src/stock_analysis/tableau/export.py` exports existing medallion tables and a Hyper mart, but there are no Tableau tables for cashflows, snapshots, realized performance, or historical recommendations.
- `src/stock_analysis/cli.py` has no command to register a deposit, list cashflows, import a snapshot, or sync recommendation results to a database.
- There is no Supabase config, dependency, migration, repository layer, or runbook.

## Key Design Decisions

- Backtesting keeps the monthly deposit assumption. The `BacktestConfig` and autoresearch CLI should continue accepting `monthly_deposit_amount`, `deposit_frequency_days`, and `deposit_start_date`.
- Live recommendations use actual Supabase data. The one-shot flow should not treat `monthly_deposit_amount` as live money unless the user explicitly runs a scenario mode.
- Store deposits on their real cashflow dates. Do not force deposits to month-end in the live ledger.
- Store account snapshots separately from cashflows. Deposits alone cannot determine current holdings, available cash, or actual returns.
- Prevent cash double counting. The live state builder must know whether a cashflow is already included in the latest account snapshot.
- Use the same actual cashflow dates for the SPY benchmark. Do not compare against a lump-sum SPY investment if the account had staggered deposits.
- Tableau should read from curated marts generated by the pipeline or directly from Supabase views. For this repo, the first implementation should keep the existing Hyper/CSV publish path and add Supabase-backed tables to the mart.

## Proposed Supabase Schema

Create SQL migrations under `supabase/migrations/`.

### `accounts`

One row per brokerage/account tracked by the product.

- `id uuid primary key`
- `slug text unique not null`
- `display_name text not null`
- `base_currency text not null default 'USD'`
- `benchmark_ticker text not null default 'SPY'`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

### `cashflows`

Manual or imported external account cashflows.

- `id uuid primary key`
- `account_id uuid not null references accounts(id)`
- `cashflow_date date not null`
- `settled_date date`
- `amount numeric not null`
- `currency text not null default 'USD'`
- `cashflow_type text not null`
- `source text not null default 'manual'`
- `external_ref text`
- `notes text`
- `included_in_snapshot_id uuid`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`

Sign convention: deposits are positive when money enters the account, withdrawals are negative when money leaves the account. Performance code converts these to investor cashflows for XIRR.

Initial `cashflow_type` values:

- `deposit`
- `withdrawal`
- `dividend`
- `interest`
- `fee`
- `tax`
- `transfer`

### `portfolio_snapshots`

Account-level valuation snapshots.

- `id uuid primary key`
- `account_id uuid not null references accounts(id)`
- `snapshot_date date not null`
- `market_value numeric not null`
- `cash_balance numeric not null`
- `total_value numeric not null`
- `currency text not null default 'USD'`
- `source text not null default 'manual'`
- `created_at timestamptz not null default now()`

### `holding_snapshots`

Ticker-level holdings for each portfolio snapshot.

- `id uuid primary key`
- `snapshot_id uuid not null references portfolio_snapshots(id)`
- `ticker text not null`
- `quantity numeric`
- `market_value numeric not null`
- `price numeric`
- `currency text not null default 'USD'`

### `recommendation_runs`

Run-level recommendation history.

- `id uuid primary key`
- `account_id uuid not null references accounts(id)`
- `run_id text unique not null`
- `as_of_date date not null`
- `data_as_of_date date not null`
- `model_version text not null`
- `ml_score_scale numeric not null`
- `config_hash text not null`
- `status text not null default 'completed'`
- `created_at timestamptz not null default now()`

### `recommendation_lines`

Ticker-level output from `portfolio_recommendations`.

- `id uuid primary key`
- `recommendation_run_id uuid not null references recommendation_runs(id)`
- `ticker text not null`
- `security text`
- `gics_sector text`
- `current_weight numeric`
- `target_weight numeric`
- `trade_weight numeric`
- `trade_notional numeric`
- `commission_amount numeric`
- `cash_required_weight numeric`
- `cash_released_weight numeric`
- `deposit_used_amount numeric`
- `cash_after_trade_amount numeric`
- `action text`
- `reason_code text`
- `expected_return numeric`
- `volatility numeric`

### `performance_snapshots`

Materialized account and benchmark return history for Tableau.

- `id uuid primary key`
- `account_id uuid not null references accounts(id)`
- `as_of_date date not null`
- `account_total_value numeric not null`
- `total_deposits numeric not null`
- `net_external_cashflow numeric not null`
- `account_time_weighted_return numeric`
- `account_money_weighted_return numeric`
- `spy_same_cashflow_value numeric`
- `spy_time_weighted_return numeric`
- `spy_money_weighted_return numeric`
- `active_value numeric`
- `active_return numeric`
- `created_at timestamptz not null default now()`

## Implementation Steps

1. Add Supabase configuration and SQL migrations.
2. Add a repository interface and a Supabase implementation.
3. Add CLI commands to register and list cashflows.
4. Add snapshot import support for account-level and ticker-level holdings.
5. Implement `build_live_portfolio_state(account_id, as_of_date)`:
   - Load the latest portfolio snapshot on or before `as_of_date`.
   - Load settled cashflows after that snapshot that are not marked as included.
   - Add unapplied net deposits to cash, not to holdings.
   - Return `PortfolioState`, `pending_contribution_amount`, and trace metadata.
6. Change `run-one-shot` so live mode uses the live state result instead of `contributions.monthly_deposit_amount`.
7. Keep autoresearch and backtesting unchanged except for naming/documentation clarity.
8. Persist recommendation runs and recommendation lines to Supabase after successful one-shot runs.
9. Compute materialized performance snapshots using actual cashflows and the same-cashflow SPY benchmark.
10. Extend Tableau exports with cashflows, performance snapshots, and recommendation history.
11. Update runbooks with exact commands and operational timing.

## Recommendation Timing

The product should generate recommendations on two triggers:

- Scheduled run after the market close, once daily market data is available.
- Event-driven run after a new deposit or withdrawal is registered and the account snapshot is updated.

The critical rule is that a recommendation should be tied to a concrete `data_as_of_date`. If a deposit is registered during market hours but the holdings snapshot or prices are stale, the product should mark the deposit as pending and produce the next recommendation after the next reliable data refresh.

## Tableau Strategy

Short term:

- Keep Tableau as a Hyper/CSV consumer produced by this repo.
- Add separate Tableau tables for `cashflows`, `performance_snapshots`, `recommendation_runs`, and `recommendation_lines`.
- Publish the Hyper datasource after each successful one-shot run.

Medium term:

- Use Supabase views for stable Tableau contracts if Tableau connects directly to Postgres.
- Keep the Python pipeline as the materialization layer for model outputs and benchmark calculations.

Avoid using Tableau as the deposit-entry interface. Use CLI first, then a small internal form or Supabase-authenticated app if manual entry becomes frequent.

## Critical Product Risks

- Supabase is not a broker or portfolio accounting engine. Without reliable holdings snapshots, return tracking will be approximate.
- Manual deposit entry can drift from brokerage reality. Snapshot import is as important as deposit registration.
- If a snapshot already includes a deposit and the live flow also adds that deposit from the ledger, recommendations will overstate available cash.
- Same-cashflow SPY benchmarking must use actual dated deposits. A simple SPY CAGR comparison is misleading for irregular deposits.
- Recommendation quality and realized return tracking are different problems. The product should eventually store executions so it can compare recommended trades against actual trades.
- Service role keys must never be exposed to Tableau workbooks, browser code, or committed configs.

## Acceptance Criteria

- A user can register a deposit dated any calendar day without editing YAML.
- Backtests and autoresearch still support the monthly deposit assumption and do not require Supabase.
- Live `run-one-shot` can use Supabase actual cashflows and snapshots.
- The best model in the main flow remains `e8-scale-0p5-contribution-aware-v1`.
- Recommendation outputs are persisted to Supabase with run-level metadata and ticker-level lines.
- Tableau can show deposits, portfolio value, active return versus SPY, and the latest recommendations.
- No Supabase secrets are committed to the repository.
